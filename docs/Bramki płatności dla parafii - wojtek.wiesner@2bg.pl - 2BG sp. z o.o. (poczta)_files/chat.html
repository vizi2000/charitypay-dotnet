1. Key Features of the Flow

  1. Centralized Registration Experience:

    • Users (organizations) registering on your platform will fill out a single form.
    • The form will prefill certain fields using data pulled from the Fiserv/Polcard API or synchronized backend systems.
    • This creates the impression that the process is fully hosted by your platform.

  2. Data Synchronization via Fiserv API:

    • Required data fields can include:
      • Organization Name
      • Address
      • Tax Identification Number (NIP/REGON)
      • Bank Account Details
      • Payment Preferences
      • KYC Documents (like ID proofs, AML compliance forms)
    • Fiserv likely provides specific API endpoints for dynamically adding sub-merchants in the Payment Facilitator (PF) model.

  3. Payment Flow Within Your Platform:

    • Users select payment amounts/methods (BLIK, Apple Pay, etc.) from your platform.
    • They are redirected seamlessly to Fiserv-hosted payment pages for transaction completion.
    • Upon transaction success/failure, Fiserv sends webhook notifications back to your system.

  4. Sub-Merchant Management with Fiserv:

    • If using the Payment Facilitator model:
      • New sub-merchants (e.g., parishes) are dynamically registered through API.
      • Your system will manage onboarding and KYC/AML compliance as part of the integration.
    • Alternatively, individual contracts per organization are an option, but the cost and management burden may increase.

2. Advantages of Proposed Integration

  • Unified Experience: The organization perceives the platform as the singular entity handling all touchpoints.
  • Reduced Redundancy: Data is only entered once and forwarded to Fiserv seamlessly.
  • Compliance Assurance: Fiserv ensures KYC/AML and regulatory compliance at the backend.
  • Scalability: Dynamic onboarding of sub-merchants through API streamlines the process when scaling up (e.g., adding new parishes).

3. Key Technical Steps

  1. Onboarding Integration:

    • Connect your registration flow to Fiserv APIs for real-time data submission.
    • Use webhooks for Fiserv feedback, such as approval or additional document requirements for KYC.

  2. API Endpoints (Example Needs):

    • Merchant Registration: Add new organizations.
    • Sub-Merchant Verification: Submit and verify KYC/AML documents.
    • Transaction Management: Create payment links, manage refunds, and chargebacks.
    • Reporting and Monitoring: Pull transactional and compliance data.

  3. Notifications and Updates:

    • Develop webhook listeners to process and act on asynchronous notifications from Fiserv (e.g., transaction status, chargeback alerts).

4. Challenges and Considerations

  • Cost Implications:
    • Initial and recurring costs for payment network certifications, PCI DSS compliance, and API maintenance.
  • Compliance Obligations:
    • Ensure documentation (AML, SPI licenses, etc.) aligns with Fiserv requirements.
  • Fraud Responsibility Under PFac Model:
    • The platform assumes liability for fraudulent transactions but can mitigate risks using Fiservs monitoring tools.

Opis Projektu Integracji Patnoci z Polcard

1. Zakres Projektu:

  • Zapewnienie rejestracji organizacji w jednym miejscu (Twoja platforma), bez koniecznoci dodatkowego logowania na inne systemy.
  • Dane rejestracyjne i KYC (Know Your Customer) wymagane przez Polcard bd automatycznie przesyane do API Polcard po ich wypenieniu na Twojej platformie.
  • Transakcje patnicze (np. darowizny) bd przetwarzane przez Polcard, ale uytkownicy kocowi widz spjny branding Twojego serwisu.

2. Gwne Etapy Integracji:

  1. Formularz Rejestracji na Twojej Platformie:

    • Pola inicjalizujce: Nazwa organizacji, adres, NIP, dane kontaktowe, itp.
    • Dodatkowe pola wymagane przez Polcard, takie jak dokumenty tosamoci, wycigi bankowe.
    • Moliwo uploadowania zacznikw np. dokumentw KYC.
    • Uwierzytelnienie uytkownika na Twojej platformie w celu ochrony danych.

  2. Integracja z API Polcard:

    • Przy uyciu API Polcard za pomoc secure tokenization przesyane s dane klienta do zarejestrowania danej organizacji jako merchant.
    • Fiserv przesya zwrotk (webhooki) z wynikiem rejestracji: status pomylny/odrzucony, potrzeba dodatkowych dokumentw itp.

  3. Przepyw Patnoci:

    • Transakcje patnicze (np. skadanie darowizn) za porednictwem Polcard bd inicjalizowane z Twojej platformy:
      • Generowanie linkw patnoci/checkout.
      • Obsuga metod patnoci: karty, BLIK, Apple Pay, itp.
    • Fiserv zarzdza transakcj (autoryzacja, rozliczenie), a dane zwrotne (np. potwierdzenie patnoci) przesyane s na Twj backend.

  4. Obsuga RCP (Reconciliation/Settlement Reports):

    • Pobieranie raportw transakcji przez API w celu uzgodnienia rozlicze na platformie.

3. Technologie i Zasoby:

  • Backend: Integracja z API Polcard (np. REST API) z odpowiednimi endpointami do rejestracji merchantw i patnoci.
  • Frontend: Formularz rejestracji kompatybilny z Twoim UX/UI i powizany z backendem.
  • Zabezpieczenia:
    • TLS/SSL dla przesyu danych.
    • Tokenizacja wraliwych informacji.
    • Zgodno z PCI DSS.

4. Wyzwania i Rozwizania:

  • Zgodno z regulacjami AML/KYC: Automatyzacja procesu poprzez API.
  • Niezauwaalno dla uytkownika: Layout formularza i branding platformy utrzymuj wraenie, e rejestracja odbywa si w jednym systemie.
  • Bdy i wymagania API: Przygotowanie fallbackw dla brakujcych/odrzuconych danych.
